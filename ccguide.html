The Beginner’s Guide to Custom Commands

So, you’re here to learn how Custom Commands work. Well, you’ve come to the right place! In this guide, we’ll be teaching the basics of creating Custom Commands and go over the common variables in them. You’ll learn, among other things, how to make a say command, embed announcement messages, build an 8-Ball, and send direct messages to users using Dynobot.

Before we begin, let’s talk about some of the syntax and assumptions we will be using in this guide.

Syntax

This guide assumes you are using the default prefix for commands, which is the question mark (?). If you have a different prefix, you should use that instead. It is also assumed you are using the regular Dynobot and not the Premium version. Both bots will still be able to use the Custom Commands and variables noted in the guide. The only difference between the two that will affect this guide is that Premium’s dashboard allows you to directly edit Custom Commands added.

Command examples will be written like this:

?command [input] - Usage and Syntax

Command Content

Chat Output (If Applicable) and Explanation


Let’s get started!

Variables:

Variables are simply placeholders and tags that you can use inside Custom Commands. These placeholders will be updated with specific values when the command is called. There are a lot of variables that can be used in Custom Commands, but a select few tend to get used the most.

{user}

Outputs a mention of the person calling the command. It’s a great, easy-to-use variable that you can include in most commands. Here’s a simple example- we’ll go more into depth on what you can do with this variable later.

?greet - Greets the user.

Hi, {user}!

Chat Output: Hi, @TheRoboticon!

When this command is called, Dynobot replaces the {user} variable with the name of the user who calls it. Again, there’s a lot more you can do with this variable that we’ll explain further on.

{server} and {channel}

Outputs the name of the server and the name of the text channel in which the command is used, respectively. Of the two, you’ll be using {channel} the most, simply because it’s more specific and has a larger range of uses.

?location - Outputs the text channel and server.

You are in the {channel} channel of {server}.

Chat Output: You are in the #general channel of Strand.

Similar to the {user} variable, the {channel} and {server} variables are updated by Dynobot when the command is called. You would get this example output if you ran this command in the #general channel of a server called Strand.

{&role}

Mentions a specified role. Useful for making alerts and notifications.

?update - Sends a message about updates mentioning Subscribers role.

{&Subscribers}, there's a new update in the {#updates} channel!

Chat Output: @Subscribers, there’s a new update in the #updates channel!

As you can see, the command outputs a mention for the Subscribers role. You might have noticed the {#updates} inside the command, which created a link to the #updates channel. When you want to post a channel inside a command, you’ll want to use that format.

{delete}

If you’ve been following along, you might have noticed that when you use these commands, it doesn’t delete the message. The {delete} tag removes the command for you, which makes your commands look cleaner.

Let’s update the greet command we used before with {delete} tag.

?delgreet - Greets the user and deletes the message calling the command.

{delete}
Hi, {user}!

Chat Output: Hi, @TheRoboticon!

You’ll see the same chat output as the ?greet command we used before, but now, the command you used to show it will get deleted! Cool, right? From now on, we’ll be including {delete} in most of our commands, since it’s so useful.

{require:role} and {require:#channel}

Adds a required role and channel to call the command, respectively. You can allow multiple roles to call the command if you specified multiple ones. This variable is great for protecting your Custom Commands that do important things.

Let’s update the ?update command we used early with a requirement that the user needs to have either the Trusted role or be a bot moderator, and that the command needs to be called in the #announcements channel.

?requpdate - Sends a message about updates mentioning Subscribers role, with channel and role requirements added.

{delete}
{require:Trusted}
{require:serverMod}
{require:#announcements}
{&Subscribers}, there's a new update in the {#updates} channel!

Chat Output: @Subscribers, there’s a new update in the #updates channel!

You’ll get the same output as before if you ran the command in the #announcements channel and have the Trusted role or if you’re a bot moderator, but if you aren’t, the command wouldn’t be run at all! It’ll be treated like a normal message and will completely be ignored by Dynobot.

What’s a bot moderator, you ask? Well, Dynobot also allows you to use two other requirements besides the roles you’ve set up on your server. Those are {require:serverMod}, which you just saw, and {require:serverAdmin}. You can add bot moderators with Dynobot using the built-in command ?addmod [user/role] and view a list of moderators with ?listmods. As for admins, Dynobot considers anyone with Administrative permissions (anyone with the Manage Server permission) to be an admin. You can use these two built-in requirements in your Custom Commands to prevent the average user from using them.

As for the {require:#channel} tag, it works exactly like how you would expect. You can use multiple instances of it to allow the Custom Command to be used in multiple channels.

$N and $N+

These variables are the essential building blocks for complex commands. $N allows you to specify inputs in your commands, using the format $1, $2, $3, and so on. $N+ works the same way, but gets everything after the specified input.

Since these variables are pretty complex, we’ll be using multiple examples for this one.

?eat [food] - Displays a message saying the user ate a specified food.

{delete}{user} ate some $1!

Chat Output: @TheRoboticon ate some bacon!

This is the output I would get if I ran the command as ?eat Bacon. The $1 acts just like the variables you used before, but instead of Dynobot replacing the variable, you get to specify it now. However, if you didn’t specify the input- for example, you simply typed ?bacon - then $1 would become undefined and you would get this output @TheRoboticon ate some undefined!, which obviously isn’t that useful.

If you wanted to use more inputs, you would simply add on $2, $3, and so forth. Let’s say you’re really hungry.

?stuff [food] [second food] - Displays a message saying the user stuffed themselves full of two specified foods.

{delete}
{user} stuffed themselves full of $1 and $2!

Chat Output: @TheRoboticon stuffed themselves full of Bacon and Eggs!

This would be the output if the calling command was ?stuff Bacon Eggs, since Dyno replaces the first variable ($1) with the input Bacon and the second variable ($2) with the input Eggs.

What if you don’t want a specified amount of inputs? This is where $N+ comes into play. We’ll use the classic “say” command to demonstrate this.

?say [message] - Makes Dyno say a message.

{delete}
$1+

Chat Output: This is a cool message!

This message would be generated by calling ?say This is a cool message! in the chat. The use of the say command should be pretty obvious- you’ll just include the message, however long it might be, after the ?say. The $N+ allows you to grab the Nth input and everything after that, which is great for Custom Commands for announcement messages. For example, $2+ would grab everything after the second input.

Now, let’s take a look at how we can use default Dynobot commands in Custom Commands.

{!command}

Calls a default Dynobot command. This is the variable is the gateway to creating complex and advanced Custom Commands that perform multiple functions in your server. Here’s an example of what you can achieve with it.

?embed [message] - Embeds the message in the current channel.

{delete}
{!announce {channel} $1+}

Chat Output: This is an embedded message!

As you can see in this command, we’ve combined four different things we’ve learned so far into one custom command. The {delete} removes the message that calls the command, the {!announce} calls the built-in ?announce command, {channel} specifies the current channel and the $1+ gets all of the text inputted. The chat output is the result of calling ?embed This is an embedded message!.

A cool trick you can do with embedded messages is create hyperlinks. Using the format [link name](link), you can place hyperlinks into your embeds. For example, using ?embed [Google](https://www.google.com) would post a hyperlink named Google.

{dm} and {dm:user}

These two variables will send a direct message to the command user or a specified user with everything after the {dm} variable. Let’s make a simple DM command to send direct messages!

?dm [user] [message] - Sends a direct message to the specified user. Requires moderator permissions.

{delete}
{require:serverMod}
{dm:$1}$2+

DM Output: Hello there!

This would be the result of calling the command as ?dm TheRoboticon Hello there!. If you wanted to make a command that DM’s the user calling it, you would simply write {dm} without specifying a user.

{choose} and {choice}

These two variables work together to create randomization in Custom Commands! {choose} takes the format {choose:option1;option2;option3} and {choice} gets a random selection from the options in {choose}. In this example, we’ll be making the classic 8ball command.

?8ball [question] - Replies to a question with an Magic 8 Ball answer.

{delete}
{choose:It is certain;It is decidedly so;Without a doubt;Yes, definitely;You may rely on it;As I see it, yes;Most likely;Outlook good;Yes; Signs point to yes;Reply hazy try again;Ask again later;Better not tell you now;Cannot predict now;Concentrate and ask again;Don't count on it;My reply is no;My sources say no;Outlook not so good;Very doubtful}
{!announce {channel} <:8ball_blob:349679658033348618>  __** Magic 8 Ball  **__
Q: $1+
A: {choice}}

Chat Output:
Magic 8 Ball
Q: Are Custom Commands fun?
A: Yes, definitely

This would be one of the possible responses of the bot when the command ?8ball Are Custom Commands fun? is called. The {choice} variable is replaced with a random element in the {choose:} list, in this case the “Yes, definitely”. When making your own commands, make sure to use semicolons to separate each element, not commas!

{silent}

This tag silences the default responses of Dynobot. Let’s take a look at using it to hide the response “Changed roles for user” when adding or removing roles.

?certify [user] - Gives the user the Certified role.

{delete}
{silent}
{!role $1 +Certified}
$1 has been certified!

Chat Output: TheRoboticon has been certified!

While the message that the user has been certified is posted, the default message Dynobot posts when roles are changed is hidden. Even though {silent} is a great tag for hiding role changes and other responses, it can interfere with some commands- for example, {silent} will prevent the {!announce} command from posting.

{respond:#channel}

Specifies a different channel to post a message, rather than the current one. We’ll use this to improve the say command.

?modsay [channel] [message] - Post a message to a specified channel. Requires moderator permissions.

{delete}
{require:serverMod}
{channel:$1}
$2+

Chat Output: Hello there!

This chat output might be the result of a user with moderator permissions calling ?modsay #general Hello there!, which would post into #general. This tag is useful for setting up logs that post into a different channel, but it will direct any text in the Custom Command there, meaning that you can’t have multiple {respond:#channel} tags.

{time}, {time12}, {date}, {datetime}, and {datetime12}

These five variables post the 24-hour time, 12-hour time, date, date and 24-hour time, and date and 12-hour time, respectively. All of them will have the timezones in EST, but if you have the Dyno V4 Dashboard, you can change the timezone displayed.

?time - Posts the date and 12-hour time.

{delete}
Here's the date and time: {datetime12}.

Chat Output: Here’s the date and time: Aug 25, 2018 03:37:53 pm.

Each variable prints a different response accordingly. This result would display if the command was called on Aug 25, 2018 at 03:37:53 pm.

{noeveryone}

Changes the @everyone mention in a command to plain text. Great to include in a say command to prevent people from abusing it.

?hug [user] - Hugs a user.

{delete}
{noeveryone}
{user} hugged $1.

Chat Output: @TheRoboticon hugged everyone.

This would be the output if I used ?hug @everyone, since the message posts without a mention. It’s worth noting that the command will still ping everyone as it’s used, since Dyno can’t instantly delete a message. However, the @everyone will become plaintext instead.

These aren’t all the variables available for Dynobot. If you’re curious, you can find more on the Dynobot server or here.

Building your Custom Commands:

Building your own Custom Commands can be really fun! However, there are some requirements and limitations for making them. Here’s how to add Custom Commands to your server.

First, head to the Dynobot Dashboard and log in. Select your server from the tab in the upper right hand corner, then go to the modules on the left side. Select and enable Custom Commands if you haven’t already, then refresh your page. From there, type your command name into first box and your code into the second, making sure you follow the guidelines below.

  • Don’t include capitals, periods, or spaces in a Custom Command name.
  • A Custom Command cannot be named an existing command name.
  • Variables like {delete} and {silent} should have their own lines.
  • Make sure that Custom Commands that include things like {!ban} and {!mute} have requirements to prevent everyone from using them.

From there, press Add Command. It make take a few seconds for the command to update to your server. You can remove commands by going to the Commands tab on the page and pressing delete to remove it.
